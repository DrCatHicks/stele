# Repo Orientation: stele

> Generated by /orient:orient. Re-run to update.

## One-line purpose
A bespoke survey engine and analytical data warehouse for longitudinal research, built to make methodological commitments (version immutability, missing-data semantics, PII routing) explicit and enforced rather than left to convention.

## Primary language(s)
Python 3.12 (API, ETL orchestration, tests), SQL/dbt (warehouse transformations), TypeScript (respondent-facing frontend)

## Pipeline / workflow stages
1. **Authoring** ‚Äî researcher drafts a survey as SurveyJS JSON via the admin frontend; the definition is stored as a mutable draft in `app.survey_definitions`
2. **Publish gate** ‚Äî on publish, `api/survey_engine/service.py:publish` runs four checks in sequence: schema validation ‚Üí lint (duplicates, dangling `visibleIf`, missing matrix row IDs) ‚Üí headless round-trip (`api/survey_engine/round_trip.py`, Node subprocess) ‚Üí SHA-256 hash+freeze; failure at any stage blocks publication
3. **Response collection** ‚Äî respondent loads survey via `/s/<short-code>`; the SurveyJS frontend captures answers + the shown-question set + the definition hash; submits to the API
4. **Raw append** ‚Äî API writes one append-only row to `app.raw_responses`: full payload, shown-set, and a frozen `definition_snapshot` (the published JSON + hash + timestamp); this is the sole ETL source
5. **Operational read-model** ‚Äî API derives `app.responses` + `app.response_items` from the same in-memory payload in the same transaction; these are rebuildable and never read by dbt
6. **PII routing** ‚Äî at submit time, high-risk free-text answers are copied to `pii.free_text_responses`; reviewers screen them via the admin console; decisions land in `pii.free_text_review_decisions`
7. **dbt ETL (on demand)** ‚Äî `dbt build --full-refresh` reads only `app.raw_responses` ‚Üí staging views ‚Üí intermediate tables ‚Üí materialized star-schema marts; no other `app.*` table is touched
8. **Analysis** ‚Äî `stele_analyst` role reads `marts` schema via SQL/R/Python; `stele_pii_reviewer` reads `pii` schema; neither role can reach the other's data

## Core concepts

**`raw_responses` as the sole ETL source**: `app.raw_responses` is append-only and is the only table dbt reads. Every response row stores the respondent's answers plus a frozen `definition_snapshot` (the published JSON + hash + timestamp), so the entire warehouse can be rebuilt from this one table. `scripts/check_invariants.py` enforces that dbt SQL never references any other `app.*` table ‚Äî this is the most architecturally significant invariant.

**Three missing-data states**: Every question √ó respondent cell has one of three states: `was_shown = true` with a value (answered), `was_shown = true` with null (shown but skipped), or `was_shown = false` (routed past by branching logic and never rendered). The `shown_questions` list is captured by the SurveyJS frontend at submit time and stored verbatim ‚Äî it is never reconstructed in SQL. Collapsing `was_shown = false` and `was_shown = true, value null` into "missing" is explicitly a bug.

**Publish gate**: Publishing is not a save. It runs four checks in sequence ‚Äî schema validation, lint, headless round-trip (Node subprocess exercises all enumerable branches), hash+freeze ‚Äî and any failure blocks publication. A question type not listed in the `frozenset` constants in `validation.py` is rejected at publish, so an unsupported type can never reach the warehouse.

**Star schema grain**: `fact_response_item` is at *selection* grain: one row per option chosen. A multi-select with three items chosen produces three rows. `fact_response` is at *respondent-question* grain: one row per person per question. Conflating them produces inflated respondent counts ‚Äî use `COUNT(DISTINCT respondent_id)` or join through `fact_response` when counting people.

**Cross-version equivalence as explicit opt-in**: `question_id` pools a question across versions of the same survey by `(survey_id, stable_name)`. Pooling across a name change or across different surveys requires explicitly setting `parent_question_id` with a written `parent_question_rationale`. The system rejects the link without the rationale. Today those columns are null; the enforcement guard in `dbt/tests/singular/` is already written.

## Common gotchas

**Dev superuser hides role/grant bugs**: The dev container runs under a Postgres superuser that bypasses all `GRANT` restrictions. CI runs the `prod-bootstrap-sim` job under the actual `stele_analyst` / `stele_etl` roles. If tests pass locally but CI fails, check `.devcontainer/postgres-init/02-schemas-and-grants.sql` for missing grants first.

**`definition_snapshot` is the tombstone discriminator, not `payload`**: The withdrawal workflow nulls `payload`, `shown_questions`, `client_metadata`, and `definition_snapshot` together. dbt's `stg_raw_responses` filters on `where definition_snapshot is not null`. Using `payload is not null` would incorrectly include tombstoned rows because `payload` and `definition_snapshot` are always nulled together ‚Äî the key is that `definition_snapshot is not null` is the canonical signal that no data has been erased.

**Free-text values are usually redacted in the analyst schema**: Free-text questions default to `pii_risk = 'high'`. In `fact_response_item`, `value_text` will be null and `value_text_redacted = true` for those rows unless a PII reviewer has explicitly promoted the response. If a query against `value_text` returns all nulls for a text question, this is why ‚Äî check `value_text_redacted` first.
