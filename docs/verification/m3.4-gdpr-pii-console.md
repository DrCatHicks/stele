# M3.4 GDPR / PII operations console — verification

Two role-gated consoles on the M3.3 admin shell, plus the dbt path that carries a
reviewer's promotion through to the analyst marts (design-doc §3.8/§3.9/§3.10).
Builds on the `m2-pii-slice` seed + warehouse — run that first.

## Roles and gates

| Console | Role | Endpoints |
|---|---|---|
| GDPR / erasure | `admin` | `GET /admin/withdrawals`; trigger is `POST /respondents/{id}/withdrawal` (M2.2) |
| Free-text PII review | `reviewer` | `GET /admin/pii/free-text?status=`; `POST .../{id}/promote`; `POST .../{id}/reject` |

Gating is covered by `api/tests/test_rbac.py`: every route is 401 without a
session, 403 for the wrong role (incl. **admin** on the reviewer routes — the
admin is not a screener, design §3.10), and the real status once the gate clears.

## 1. Promotion data model (no PII in the decision)

`pii.free_text_review_decisions` records the per-response decision keyed by
`(raw_response_id, question_name)` — `status ∈ {promoted, rejected}`,
`reviewed_by`, `reviewed_at`, `note`. It holds **no `value_text`**: a "pending"
answer is a `pii.free_text_responses` row with no decision (the queue LEFT JOINs
the two). This separation lets dbt read the decision without reading the PII text.

## 2. Trust boundary — ETL reads decisions, never PII text

The migration grants `stele_etl` USAGE on `pii` + SELECT on the decision table
alone:

```bash
PGPASSWORD=dev psql -h localhost -U stele_etl -d stele -c \
  "select count(*) from pii.free_text_review_decisions;"     # OK
PGPASSWORD=dev psql -h localhost -U stele_etl -d stele -c \
  "select count(*) from pii.free_text_responses;"            # permission denied
```

The decision table is a dbt source under the `pii` schema —
`scripts/check_invariants.py` permits it (invariant 1/4 is app-scoped: it carries
no response *content*), and invariant 6 stays bound because the `value_text` CASE
still references `pii_risk`.

## 3. Promotion round-trip (API → pii → dbt → marts)

The seed promotes one high-risk answer (R1's `ft_high`). After `dbt build`:

```bash
PGPASSWORD=dev psql -h localhost -U stele_etl -d stele -c "
  SELECT dq.stable_name, fri.value_text, fri.value_text_redacted
  FROM marts.fact_response_item fri
  JOIN marts.dim_question dq ON dq.question_id = fri.question_id
  WHERE dq.stable_name = 'ft_high' ORDER BY fri.value_text NULLS LAST;"
```

Expected on a clean seed (4 respondents): the promoted answer surfaces its
`value_text` with `value_text_redacted = f`; the other high-risk `ft_high` rows
(unpromoted, shown-skipped, routed-past) stay `null` / `t`. So vs. M2's all-
redacted `ft_high`: marts now show **4** `value_text` rows and **3** redacted.

`api/tests/test_pii_review.py` covers the API behaviour (pending queue carries
the text, promote idempotency, reject→re-promote, 404). The dbt singular test
`promoted_free_text_in_marts` proves the marts side; `free_text_redaction_parity`
was extended to expect promoted high-risk rows surfaced rather than redacted.

## 4. Guardrails

- `python3 scripts/check_invariants.py` clean (incl. the new `pii` source in
  invariant-4-sources-yaml and the gated CASE in invariant 6).
- `dbt build` runs `promoted_free_text_in_marts` + the extended
  `free_text_redaction_parity`.
- Migration `c7d8e9f0a1b2` down/up round-trips (downgrade drops the table and
  revokes the `stele_etl` grants).
