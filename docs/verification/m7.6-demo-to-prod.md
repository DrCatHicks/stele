# M7.6 — External DB access, demo seed, secret rotation, and the demo→prod checklist

The last M7 story. It closes the operable gaps for the client demo and consolidates
the "revisit before real production" items that earlier stories deliberately
deferred. Three things ship as code; the rest is the checklist below.

| Ships | Where |
|---|---|
| Opt-in public Postgres endpoint for analyst/reviewer direct access | `infra/railway/` — `railway_tcp_proxy`, `var.enable_postgres_proxy` |
| `seed` entrypoint verb (initial admin operator, idempotent) | `scripts/docker-entrypoint.sh` |
| Deployment-secret rotation helper + tofu override path | `scripts/rotate_role_password.py`, `infra/railway/variables.tf` |

Module apply steps and the state-holds-secrets caveat stay in `infra/README.md`;
the deploy model and the admin-connection tradeoff stay in
`docs/verification/m7.4-railway.md`.

## Bringing up the demo

1. **Apply the module** (`infra/README.md` § apply). The web service's `migrate`
   pre-deploy bootstraps the four `stele_*` roles + schemas + grants and runs
   `alembic upgrade head` as the admin identity, before the web process starts.
2. **Seed the initial admin** — a one-off *inside* the running web container. This
   needs the Railway CLI (`infra/README.md` § Operating the deployment). Use
   `railway ssh`, not `railway run`: `run` executes on your laptop with env injected
   (so the in-image entrypoint isn't there and the private-network Postgres is
   unreachable); `ssh` runs in the container, which already has `STELE_DATABASE_URL`
   set to the `stele_api` role — so you supply only the admin credentials:
   ```bash
   railway ssh --service web
   # then, at the in-container shell:
   STELE_ADMIN_EMAIL=you@example.com STELE_ADMIN_PASSWORD='…' \
     bash /usr/local/bin/docker-entrypoint.sh seed
   ```
   The `seed` verb runs `scripts/bootstrap_admin.py` as least-privilege `stele_api`
   (it only INSERTs into `app.users`) and is idempotent — an existing admin is left
   untouched. Success prints `Created admin <email> (id …)`.
3. **Log in** to the web URL (`tofu output web_url`) and author surveys in the UI.

> **Why admin-only — no example surveys.** `seed_example_survey.py` writes synthetic
> respondents into `app.raw_responses`, which is **append-only and the sole ETL
> source** (invariant 1). Synthetic responses in a real instance are permanent
> warehouse pollution — only the tombstone workflow ever touches `raw_responses`.
> So the prod seed path establishes only the admin login; the same `seed` verb is
> used in the demo and in prod. To show a sample survey during the demo, import a
> validated `patterns/*.json` fixture through the editor — no synthetic responses.

## Analyst / reviewer direct database access

Analysts and reviewers query the warehouse directly with the per-user login roles
the M3.5 provisioning CLI mints (`scripts/provision_db_credential.py`), not through
the API. On Railway, Postgres is on the project's private network, so those
credentials need a public path in.

1. **Enable the proxy** — set `enable_postgres_proxy = true` and apply. Get the
   public endpoint:
   ```bash
   tofu output postgres_proxy        # → host:port  (null when disabled)
   ```
2. **Provision a credential** over the admin connection (the proxy endpoint or a
   `railway ssh --service web` shell), then hand the analyst their host/port +
   one-time password:
   ```bash
   STELE_PROVISION_DATABASE_URL=… \
     uv run python scripts/provision_db_credential.py provision \
       --access analyst --subject jdoe@example.com
   ```
   They connect to the proxy host:port and run `SET ROLE stele_analyst;` (the login
   role is NOINHERIT — privilege-less until the explicit `SET ROLE`).

> **It is a public endpoint.** Auth is Postgres password + the least-privilege group
> role, but `enable_postgres_proxy` opens 5432 to the internet, so it is **off by
> default**: turn it on only while someone needs access, and treat it as a revisit
> item before real PII (below). Revoke per-person with the provisioning CLI's
> `revoke`; turn the proxy back off when direct access isn't needed.

## Secret rotation

The module **generates** every role password (`random_password`, in state). But
`bootstrap_roles.py` creates each role's password exactly once and **never
re-passwords an existing role** (M7.3, so re-deploys need no secrets). So
regenerating a tofu secret alone updates the connection strings *without* changing
the live Postgres password — auth would break. Rotation is therefore two moves, and
the override variables exist to make the second one clean.

### A `stele_*` role (api / etl / analyst / pii_reviewer)

```bash
# 1. Change the LIVE password (shown once on your terminal, not stdout):
STELE_ADMIN_DATABASE_URL=… uv run python scripts/rotate_role_password.py stele_analyst
# 2. Feed that value back so the services' connection strings match the live role:
#    set the matching override in terraform.tfvars and apply.
#      stele_analyst_password_override = "<the new password>"
tofu apply
```

A non-empty `*_password_override` wins over the generated value in `main.tf`'s
locals, so the env vars (`STELE_*_PASSWORD`, the connection-string locals, the
`DBT_PASSWORD` for the ETL cron) all carry the rotated password and the next deploy
picks it up. Leaving the override empty keeps the convenient generated-secret path.
These overrides double as the "operator-supplied secrets, never born in state" posture
for real production.

### The admin / owner role

Same two steps with `admin_password_override`, plus one caveat: the Railway Postgres
image only reads `POSTGRES_PASSWORD` at initdb, so the live admin password is changed
solely by the `ALTER ROLE` (step 1). The override keeps `STELE_ADMIN_DATABASE_URL`
(the migrate pre-deploy's identity) and `POSTGRES_PASSWORD` consistent with it.

### The session secret

No live-role coupling — rotating it just invalidates existing login cookies (forcing
re-login). Regenerate it directly:

```bash
tofu apply -replace='random_password.session_secret'
```

## Demo → production checklist

Carry these before the instance holds real respondent PII or serves more than the
demo. Each was deferred deliberately; this is the consolidated list.

- [ ] **Encrypted remote tofu state.** State holds every generated secret and is
  local + gitignored for the demo (`infra/README.md` § state holds secrets). Move to
  an encrypted remote backend so secrets aren't only on one laptop and multiple
  operators can apply. `infra/railway/versions.tf` carries a commented `backend "s3"`
  stub inside its `terraform {}` block; uncomment and fill it in, then
  `tofu init -migrate-state` to move the local state to the encrypted remote backend.
  Not wired here — there's no backend account in the sandbox (same CI-only posture as
  the rest of the module).
- [ ] **Volume backups + restore drill.** The Postgres data volume is the only
  durable store; ops metadata (`ops.etl_runs`) and PII (`pii.*`) live there. Enable
  scheduled volume backups for the `postgres` service (Railway dashboard / API — not a
  provider resource), and actually rehearse a restore. The marts are reconstructible
  from `raw_responses` via `make etl`; `raw_responses` and `pii.*` are not — they need
  a real backup.
- [ ] **Retention vs erasure.** A volume backup taken before a withdrawal still
  contains that respondent's PII, so the M2.2 tombstone erasure is not complete until
  backups predating it have aged out. Decide a backup-retention window and document it
  alongside the withdrawal workflow, so "right to erasure" has a bounded, stated
  guarantee rather than an open-ended one.
- [ ] **EU data residency.** US hosting is fine for the demo (project memory). Before
  EU PII, set `region` to a EU metal region (`europe-west4`) and confirm backups and
  any future log/observability sinks stay in-region. `region` is already the module's
  cloud-neutral knob, so this is a value change, not a rewrite.
- [ ] **Admin creds on the web service** (carried from M7.4). The web service's env
  holds `STELE_ADMIN_DATABASE_URL` because its pre-deploy `migrate` needs it; a web
  compromise could connect as DB owner. Revisit a separate single-identity migrate
  service when moving off Railway to RDS / Cloud SQL. Rotate the admin password
  independently of redeploys in the meantime (above).
- [ ] **Turn the Postgres proxy back off** when no analyst needs standing direct
  access (`enable_postgres_proxy = false`), and prefer per-person, revocable M3.5
  credentials over shared logins.

## Verification

| Criterion | How |
|---|---|
| Module well-formed + type-checks (proxy, overrides) | CI `infra` job: `tofu fmt -check -recursive infra/`, `tofu init -backend=false` + `tofu validate`. The `railway_tcp_proxy` attrs and the override locals resolve against the real provider schema. |
| `seed` verb dispatches to the admin bootstrap, from the app root | `api/tests/test_docker_entrypoint.py::test_seed_runs_admin_bootstrap_from_app_root`; CI `docker-build` runs `docker run … seed` in print mode. |
| Rotation actually changes the live password | `api/tests/test_rotate_role_password.py::test_rotate_changes_the_live_password` — ALTERs a throwaway role through the CLI, then connects *as* it with the new password (and proves the old one fails). Runs under the elevated CI connection. |
| Rotation connection precedence + role/secret handling | the env-only unit tests in the same file (admin-URL precedence, alphanumeric secret, identifier guard, secret-to-terminal-not-stdout). |

A real `tofu apply` and a live rotation against Railway are operator steps; CI
validates the module shape and the script logic, not a live deploy (same posture as
every M7 story).
