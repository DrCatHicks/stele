# M7.4 — Railway OpenTofu module + secrets

> **Superseded in part by D7** (`docs/verification/d7-prebuilt-image.md`): the web
> and ETL services no longer build from the repo — they pull a single prebuilt GHCR
> image, migrations run on web start (`STELE_MIGRATE_ON_START`) instead of a
> `railway.json` pre-deploy command, and `railway.json`/`railway.etl.json` are gone.
> The deploy-model rationale and the admin-connection tradeoff below still hold;
> read this for the *why*, the D7 doc for the *current* mechanics.

Deploys the survey engine to Railway as code: one project, a managed Postgres
service, and a web service Railway builds from the repo Dockerfile, with all
secrets generated by the module. Consumes the role-bootstrap wiring from M7.3
(`scripts/bootstrap_roles.py`, the `migrate` entrypoint).

Module: `infra/railway/`. Deploy config: repo-root `railway.json`. Apply steps and
state-secret caveat: `infra/README.md`.

## Deploy model (and why)

The provider (`terraform-community-providers/railway`) can't set a start or
pre-deploy command on a service, and `source_image` services can't read a config
file — so the web service is built from the repo (`source_repo` + `config_path`),
and `railway.json` carries what the provider can't:

```jsonc
"deploy": {
  // startCommand omitted on purpose — see the ENTRYPOINT note below.
  "preDeployCommand": "bash /usr/local/bin/docker-entrypoint.sh migrate",
  "healthcheckPath": "/api/health",
  "restartPolicyType": "ON_FAILURE"
}
```

`preDeployCommand` is the chosen migration mechanism: Railway runs it to completion
and only shifts traffic to the new release if it exits 0, so a failed migration
leaves the previous deploy serving. The alternatives were rejected for this demo:
a prebuilt-image deploy can't run a pre-deploy command (migrations become a manual,
forgettable step), and a separate one-shot migrate service gives no ordering
guarantee against web (crash-loops on schema-changing deploys).

> **ENTRYPOINT interaction.** Railway's docs are explicit: a custom start command
> *"overrides the image's `ENTRYPOINT` in exec form"* — it does **not** get appended
> as an argument the way `CMD` does. So a bare `startCommand: "web"` would try to
> exec a nonexistent `web` binary. Two consequences, both handled above:
> - **web** sets *no* `startCommand`, so Railway uses the image's own
>   `ENTRYPOINT ["docker-entrypoint.sh"]` + `CMD ["web"]` — exactly the dispatch the
>   `docker-build` CI job exercises (`docker run stele web`).
> - **migrate** has no Dockerfile default to fall back to and also overrides the
>   entrypoint, so `preDeployCommand` gives the full invocation
>   (`bash /usr/local/bin/docker-entrypoint.sh migrate`); `bash <abs-path>` is robust
>   whether Railway runs it via a shell or execs it directly.

## The admin-connection decouple

The web service's pre-deploy `migrate` needs an **admin** identity (CREATEROLE,
owns the DB) while the web **process** must stay least-privilege `stele_api`. A
service's pre-deploy command and its runtime process share one environment, so the
two identities can't both live on `STELE_DATABASE_URL`. M7.4 splits them:

| Var | Identity | Used by |
|---|---|---|
| `STELE_DATABASE_URL` | `stele_api` (least-privilege) | the web process at runtime |
| `STELE_ADMIN_DATABASE_URL` | admin / DB owner | pre-deploy `migrate` (bootstrap + alembic) |

Both `scripts/bootstrap_roles.py` and `api/alembic/env.py` prefer
`STELE_ADMIN_DATABASE_URL`, falling back to `STELE_DATABASE_URL`. Dev and CI set
only `STELE_DATABASE_URL` (one admin identity), so the fallback leaves them
unchanged; bootstrap-er still equals migrator (both read the admin var).

**Tradeoff (accepted):** the admin connection string is present in the web
service's environment, because the pre-deploy step that needs it runs in that
service. A web-app compromise could therefore read it and connect as the DB owner.
This is the realization of M7.3's design ("migrate is a verb of this image, admin
identity") and it actually narrows the runtime (the web process drops to
`stele_api`, never admin). Mitigations / revisit triggers:

- rotate the admin password independently of redeploys;
- when moving off Railway to RDS / Cloud SQL, or before PII goes real, reconsider a
  separate single-identity migrate service (no admin creds on web). This pairs with
  the EU-residency / encrypted-state items on the same before-real-prod checklist.

## Verification

| Criterion | How |
|---|---|
| Module is well-formed + type-checks | CI `infra` job: `tofu fmt -check -recursive infra/`, then `tofu init -backend=false` + `tofu validate` in `infra/railway/`. Run locally the same way. |
| Admin-URL precedence (unit) | `api/tests/test_bootstrap_roles.py::test_conninfo_prefers_admin_url` / `::test_conninfo_falls_back_to_database_url`. |
| Admin-URL precedence (end-to-end, non-superuser) | CI `prod-bootstrap-sim`: bootstrap reads `STELE_ADMIN_DATABASE_URL`; the migrate step sets **both** vars (admin + a real-but-underprivileged `stele_api`) — a green `alembic upgrade head` proves alembic prefers the admin var, since `stele_api` would be denied `CREATE TABLE`. |
| Dev/CI unchanged by the decouple | The rest of the suite still sets only `STELE_DATABASE_URL` and passes (fallback). |
| Healthcheck target exists | `GET /api/health` (see `api/tests/test_app_composition.py`). |

A real `tofu apply` is **not** run here or in CI — there's no Railway account/token
in the sandbox (same CI-only posture as the Docker image build and the role
bootstrap). The apply is an operator step; see `infra/README.md`.
