# M3.5 analyst/reviewer DB-credential provisioning — verification

Proves the design-doc §3.10 access model: analysts and reviewers connect to
Postgres **directly** (marts / pii) with their own per-person login roles, which
an admin provisions, rotates, and revokes **out of band** — never through the API.
The privileged `CREATE ROLE` / `GRANT` lives in `scripts/provision_db_credential.py`;
the app keeps only an auditable registry (`app.db_credential_grants`) and a
read-only, admin-gated view of it.

## Trust boundary — why a CLI, not an endpoint

`stele_api` has neither `CREATEROLE` nor `ADMIN OPTION` on the group roles, so it
**cannot** mint logins — by construction, not by convention:

```bash
psql "postgresql://stele_api:dev@localhost:5432/stele" -c "CREATE ROLE probe NOLOGIN;"
# ERROR:  permission denied to create role
```

Putting `CREATEROLE` behind the public request path would make the most-exposed
component the highest-value escalation target. Instead the operator runs the CLI
with an elevated connection (`STELE_PROVISION_DATABASE_URL` — a `CREATEROLE` role
with `ADMIN OPTION` on the group roles in prod; the service superuser in
dev/CI). See PR discussion for the alternatives weighed (provisioner-role-behind-API,
generate-SQL-for-admin) and why they lose on trust boundary or secret handling.

`STELE_PROVISION_DATABASE_URL` is **required**: an unset URL fails the run rather
than guessing, so a missing/misspelled var can't silently provision into the wrong
database. For local dev against the container superuser, opt into the fallback
explicitly with `STELE_ALLOW_DEV_FALLBACK=1`.

## 1. Provision (CLI)

```bash
uv run python scripts/provision_db_credential.py provision \
    --access analyst --subject jdoe@example.com
```

Creates a per-person login role `stele_<access>_<slug>_<suffix>`, grants it the
§3.3 group role, writes an `active` registry row, and shows the password **once**
(never stored). The role is `NOINHERIT`, so it carries no privilege until the user
runs `SET ROLE`. `--access reviewer` maps to `stele_pii_reviewer` instead.

The password is written to the **controlling terminal** (`/dev/tty`), not stdout, so
a redirected or captured stdout (`… > file`, a CI job, a wrapper) never receives the
secret, and the command **fails closed** when there's no terminal to show it to. For
non-interactive use, point `STELE_PROVISION_SECRET_SINK` at a writable path. Verified
in `api/tests/test_provision_cli.py` (password lands in the sink, absent from stdout).

## 2. NOINHERIT — privilege only after SET ROLE

A freshly provisioned analyst login cannot read marts until it assumes the group
role:

```bash
# As the new login role, before SET ROLE: schema access is denied.
psql "postgresql://<login>:<pw>@localhost:5432/stele" \
  -c "SELECT * FROM marts.dim_question LIMIT 1;"
# ERROR:  permission denied for schema marts

# After SET ROLE, the group's privileges apply.
psql "postgresql://<login>:<pw>@localhost:5432/stele" \
  -c "SET ROLE stele_analyst; SELECT count(*) FROM marts.dim_question;"
```

Asserted in `api/tests/test_provision_cli.py` (`rolinherit = false` + group
membership present).

## 3. Registry view (admin-only API)

`GET /admin/db-credentials` lists the registry for the admin UI. It is read-only
and never returns a password.

- unauthenticated → **401**
- researcher / reviewer → **403**
- admin → **200**, rows of `{subject_label, access, login_role, status, …}`

Covered by `api/tests/test_db_credentials.py`.

## 4. Rotate / revoke (CLI)

```bash
uv run python scripts/provision_db_credential.py rotate <login>   # new password, once
uv run python scripts/provision_db_credential.py revoke <login>   # DROP ROLE + mark revoked
uv run python scripts/provision_db_credential.py list             # full registry
```

Role DDL and the registry write commit in one transaction, so a role never exists
without its audit row (and vice versa). Revoked rows are retained (audit log);
re-provisioning the same subject yields a fresh, distinct login role. A second
*active* credential for the same `(subject, access)` is refused (partial unique
index).

## Injection safety

The subject label flows into a role identifier. `provisioning.derive_login_role`
slugifies it to `[a-z0-9_]`, validates the result against a strict, length-bounded
identifier regex (raising on an empty slug), and the CLI additionally quotes via
`psycopg.sql.Identifier`. Hostile subjects (`'); DROP TABLE …`) are covered in
`api/tests/test_db_credentials.py::test_derive_login_role_neutralizes_injection`.

## What's deliberately out of scope

Provisioning is CLI-only; there is no API *action* endpoint. A future API-driven
request flow (admin clicks "provision" in the M3.3 UI → `pending` registry row →
operator fulfils via CLI) would set `provisioned_by`, which the CLI leaves NULL
today. The `GET` endpoint is gated with M3.2's `require_role("admin")`.
