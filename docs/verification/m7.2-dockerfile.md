# M7.2 Production image — verification

One image, three entrypoints. The same artifact runs the web service, the schema
migration, and the logged ETL — selected by the first argument and dispatched by
`scripts/docker-entrypoint.sh`.

| Entrypoint | Process | Connection / secret env |
|---|---|---|
| `web` (default) | `uvicorn api.main:app` — API under `/api` + SPA at `/` | `STELE_DATABASE_URL` (stele_api), `STELE_SESSION_SECRET`, `STELE_COOKIE_SECURE` |
| `migrate` | `alembic upgrade head` (from `api/`) | `STELE_DATABASE_URL` (the admin identity that also bootstraps roles — see [[project-deployment]]) |
| `etl` | `scripts/run_etl.py` (logged `dbt build`) | `DBT_HOST`/`DBT_USER`/`DBT_PASSWORD`/`DBT_DBNAME`, `STELE_ETL_DATABASE_URL`, `GIT_SHA` |

`GIT_SHA` provenance: the image has no `.git` and no `git` binary, so the runner's
`git rev-parse` falls back to `$GIT_SHA`. Bake it at build time
(`docker build --build-arg GIT_SHA=$(git rev-parse HEAD) .`) so `ops.etl_runs`
records a real commit instead of NULL; unset, runs log `git_sha = NULL` (graceful).

## Why a Node base for a Python service

The publish gate drives synthetic respondents through the real SurveyJS engine via
a Node oracle (`frontend/scripts/roundTrip.mjs` → `survey-core`), and that runs **at
request time** whenever a researcher publishes a real-respondent survey. So the
runtime needs Node + `survey-core`, not just Python, or publishing fails closed
(503, `RoundTripUnavailable`). The image starts from `node:20-bookworm-slim` and
lets `uv` supply and manage Python 3.12 + every Python dep (including dbt) — one
base, no cross-image binary copying. The built SPA, the oracle script, and the
frontend `node_modules` (carrying `survey-core`) all land under `/app/frontend` so
`api/main.py`'s StaticFiles mount and `api/survey_engine/round_trip.py` both resolve.

## 1. Build

```bash
docker build -t stele:local .
```

Multi-stage: stage 1 (`pnpm build`) produces the SPA bundle + `node_modules`;
stage 2 installs Python deps via `uv sync --frozen`, copies `api/ dbt/ scripts/`,
then copies the SPA + oracle assets from stage 1.

## 2. Composition smoke (no DB)

`create_async_engine` is lazy, so importing the app doesn't connect — this checks
both runtimes, dbt + alembic, the SPA bundle, the oracle's `survey-core`, and that
the app imports (which also exercises the `STELE_FRONTEND_DIST` wiring):

```bash
docker run --rm --entrypoint bash stele:local -c '
  set -euo pipefail
  python --version && node --version
  uvicorn --version && alembic --version && dbt --version
  test -f /app/frontend/dist/index.html
  test -d /app/frontend/node_modules/survey-core
  test -f /app/frontend/scripts/roundTrip.mjs
  python -c "import api.main"
'
```

CI runs exactly this in the `docker-build` job (`.github/workflows/ci.yml`), so the
Dockerfile can't rot silently.

## 3. Entrypoint dispatch

The dispatcher's `STELE_ENTRYPOINT_PRINT=1` hook reports the resolved working dir +
argv instead of exec'ing — used by the unit tests and quick to eyeball:

```bash
docker run --rm -e STELE_ENTRYPOINT_PRINT=1 stele:local web      # → uvicorn api.main:app …
docker run --rm -e STELE_ENTRYPOINT_PRINT=1 stele:local migrate  # → alembic upgrade head
docker run --rm -e STELE_ENTRYPOINT_PRINT=1 stele:local etl      # → python scripts/run_etl.py
docker run --rm stele:local frobnicate ; echo $?                 # → exit 64 (EX_USAGE)
```

Pass-through args forward after the verb, e.g. `etl -- --select dim_question`.

The dispatch logic is unit-tested without Docker:

```bash
uv run pytest api/tests/test_docker_entrypoint.py
```

## 4. End-to-end against a database

Not exercised here — needs a reachable Postgres with the role/grant model applied
(M7.3 managed-PG role bootstrap + runner). The shape:

```bash
docker run --rm -e STELE_DATABASE_URL=… stele:local migrate
docker run --rm -e STELE_ETL_DATABASE_URL=… -e DBT_HOST=… stele:local etl
docker run --rm -p 8000:8000 -e STELE_DATABASE_URL=… -e STELE_SESSION_SECRET=… stele:local
```

## Known follow-ups

- The `uv` binary is pulled from `ghcr.io/astral-sh/uv:latest`; pin it to a specific
  tag once M7.4's first green build confirms a working one (`uv.lock` + `--frozen`
  already pin every actual dependency, so this is low-risk).
- ETL artifacts archive to `dbt/etl_artifacts/<run_id>/` on the container
  filesystem, which is ephemeral on Railway — `ops.etl_runs` is the source of truth;
  durable artifact storage is M7.5.
- `uvicorn` runs without explicit `--proxy-headers`; revisit when the platform proxy
  is wired in M7.4 if request-scheme detection is needed.
