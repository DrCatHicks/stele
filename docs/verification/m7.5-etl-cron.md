# M7.5 — ETL cron service + artifact ops

> **Superseded in part by D7** (`docs/verification/d7-prebuilt-image.md`): the cron
> service now pulls the same prebuilt GHCR image as web (not a repo build), selects
> the `etl` verb with `STELE_ENTRYPOINT=etl` instead of a `railway.etl.json`
> `startCommand`, and its `restartPolicyType: NEVER` is now a Railway dashboard
> setting (the provider can't express it on an image service). The cron's purpose,
> least-privilege boundary, and artifact handling below are unchanged.

Runs ETL on a schedule on Railway, and makes the run record durable despite an
ephemeral container filesystem. Builds on the M6.2/M6.3 runner (`api/etl/runner.py`
→ `scripts/run_etl.py`, the `etl` entrypoint verb) and the M7.4 Railway module.

## The cron service

`infra/railway/main.tf` adds a `railway_service "etl"`: the **same image** as web,
built from the repo, but started on the `etl` verb on a schedule
(`var.etl_cron_schedule`, default `0 6 * * *` — daily 06:00 UTC). Railway runs the
container on the cron tick and lets it exit; it connects only as least-privilege
`stele_etl` and never bootstraps roles or migrates.

It points at its **own** config file, `railway.etl.json`, not web's `railway.json`:

```jsonc
"deploy": {
  "startCommand": "bash /usr/local/bin/docker-entrypoint.sh etl",
  "restartPolicyType": "NEVER"
}
```

Three reasons it can't share web's config:

- **No healthcheck** — a cron job is not a long-running service with a `/api/health`.
- **No migrate pre-deploy** — schema changes stay web's job (the cron runs as
  `stele_etl`, which can't bootstrap or migrate).
- **`startCommand` overrides the ENTRYPOINT in exec form** (the same Railway gotcha
  M7.4's migrate pre-deploy handled), so `railway.etl.json` gives the full
  `bash /usr/local/bin/docker-entrypoint.sh etl` invocation rather than a bare verb.

`restartPolicyType: NEVER` is deliberate: a failed cron run must not crash-loop — it
re-runs on the next tick, and the failure is recorded in `ops.etl_runs`.

### Connection env (`railway_variable_collection "etl"`)

| Var | Used by | Value |
|---|---|---|
| `STELE_ETL_DATABASE_URL` | runner's `ops.etl_runs` writes (psycopg) | `stele_etl` URL |
| `DBT_HOST` / `DBT_USER` / `DBT_PASSWORD` / `DBT_DBNAME` | dbt's postgres profile | internal host / `stele_etl` / pw / `stele` |

The dbt profile's port is fixed at 5432, matching Railway's internal Postgres.
`GIT_SHA` is intentionally **not** set: Railway auto-injects
`RAILWAY_GIT_COMMIT_SHA` into repo-built services, which `runner.git_sha()` reads
(after a real git binary, then `GIT_SHA`) so the run keeps non-null provenance.

> **Build tradeoff (accepted).** The cron service builds the Dockerfile from the
> repo independently of web, so a push rebuilds the image twice. The community
> Railway provider can't share one service's build artifact with another (no
> registry handoff), and building from the repo keeps the cron on exactly the
> tested image. At demo scale the extra build minutes are cheap; revisit (a shared
> prebuilt image in a registry) alongside the other before-real-prod items.

## Artifact ops — durable run record on an ephemeral FS

The runner archives `manifest.json` / `run_results.json` to
`dbt/etl_artifacts/<run_id>/`. On a Railway cron container the filesystem is
**ephemeral** — that archive is discarded when the run exits. `ops.etl_runs` (in
managed Postgres) is therefore the durable record.

It already held run-grain metadata (timings, status, source/mart row counts, dbt
version, git sha). M7.5 adds **`dbt_run_results`** (migration `a1b2c3d4e5f6`): a
compact summary the runner parses from `run_results.json` — `elapsed_time` plus,
per node, `unique_id` / `status` / `execution_time` / `message` / `rows_affected`.
This pulls the part that makes a *failed* run debuggable (which node errored, with
what message) into the durable record, while the full manifest stays on disk only
(large, and reconstructible from the code at `git_sha`).

No new grants: `stele_etl` already holds INSERT/UPDATE on `ops.etl_runs`
(f0a1b2c3d4e5), and a column inherits the table's privileges. The column is
nullable — a build can fail before dbt emits `run_results.json` at all (e.g. a
missing binary), and capturing the summary is best-effort so it never masks the
run's real outcome.

## Verification

| Criterion | How |
|---|---|
| Module is well-formed + type-checks (incl. the cron service) | CI `infra` job: `tofu fmt -check -recursive infra/`, then `tofu init -backend=false` + `tofu validate`. Run locally the same way. |
| `cron_schedule` is a real provider attribute | `tofu validate` resolves `railway_service.etl.cron_schedule` against the provider schema. |
| Run-results summary parsing (unit) | `api/tests/test_run_etl.py::test_summarize_run_results_extracts_compact_per_node` / `::test_summarize_run_results_missing_file_returns_none`. |
| `dbt_run_results` persists end-to-end (real grants) | `::test_execute_run_success_records_row` asserts the column round-trips, connecting as `stele_etl` (so a missing grant fails). Runs in CI's `python-test` + `dbt` jobs (the latter runs the real `run_etl.py` → real `run_results.json`). |
| Railway provenance fallback | `::test_git_sha_falls_back_to_railway_env`. |
| Migration round-trips | `alembic upgrade head` then `downgrade -1` on the column. |

A real `tofu apply` is **not** run here or in CI (no Railway token in the sandbox —
same CI-only posture as the image build and role bootstrap). The cron only executes
on a live Railway deploy; CI proves the module validates and the runner writes the
durable record under real grants.
